Connecting RAG
With the Internet

Static knowledge bases go stale the moment training ends. Your LLM knows nothing about what happened last week. Here is how to wire your RAG pipeline to live internet data — and when to actually do it.

Standard RAG has a built-in clock problem. You ingest documents, embed them, store them in a vector database — and the moment you finish, the knowledge is already aging. Ask your system "what is the current price of NVIDIA stock?" and it will answer based on data from your last ingestion run, which might be weeks old. Ask it "what changed in the new OpenAI API last week?" and you will get confident hallucination.

The fix is not rebuilding your entire stack. It is adding internet retrieval as a layer on top of — or alongside — your existing pipeline. Done correctly, this makes your RAG system genuinely useful for time-sensitive queries while preserving the speed, cost, and auditability of your internal knowledge base for stable content. Done incorrectly, it adds latency, cost, and a new category of hallucination (confidently citing wrong URLs).

This post covers the architecture choices, the specific tools, and the working code to connect your RAG system to the live internet — and the routing logic that decides when to use internal retrieval versus live web search.

TL;DR
  • Static RAG goes stale. Static systems suffer 40–60% higher hallucination rates due to knowledge decay on time-sensitive queries. Any question about current prices, recent events, or latest regulations requires live data.
  • Web search APIs — Tavily, Exa, Firecrawl, Serper — are the integration layer. They return LLM-ready structured content from the live web, not raw HTML. The right choice depends on whether you need speed (Tavily), semantic understanding (Exa), or full page extraction (Firecrawl).
  • CRAG (Corrective RAG) is the most robust architecture: retrieve from internal knowledge base first, evaluate retrieval quality, fall back to web search only when internal results are insufficient.
  • Adaptive routing classifies queries by freshness requirement before retrieval — static internal retrieval for stable content, web search for time-sensitive queries. Reduces unnecessary web API calls by 60–70%.
  • Tiered freshness strategy — cache stable web results (24–72h), refresh semi-stable content weekly, always live for real-time data — controls costs while keeping latency acceptable.

The Knowledge Decay Problem

Every LLM has a training cutoff. Every static knowledge base has an ingestion timestamp. After that moment, the world keeps changing — but your system does not. This is knowledge decay, and it creates a specific failure mode: your system confidently answers time-sensitive questions with outdated information that it cannot distinguish from current facts.

Static RAG
Internal Knowledge Base
Documents ingested, chunked, embedded, and indexed. Knowledge is frozen at ingestion time. Retrieval is fast, cheap, and offline. No external dependencies at query time.
Strengths
  • Sub-100ms retrieval latency
  • No external API cost at query time
  • Fully auditable — every chunk is known
  • Works offline, no rate limits
  • Perfect for stable documents (policies, manuals, contracts)
Limitations
  • Stale on current events, prices, regulations
  • 40–60% higher hallucination on time-sensitive queries
  • Re-indexing pipeline required for updates
  • Cannot answer "what changed last week?"
Web-Connected RAG
Live Internet Retrieval
Every query potentially hits the live web via a search API. Knowledge is always current. Higher latency and cost per query. Requires quality filtering — the web contains misinformation.
Strengths
  • Always current — news, prices, new releases
  • No knowledge decay on time-sensitive queries
  • Cites verifiable sources
  • Covers topics not in your internal corpus
Limitations
  • 500ms–1s+ added latency per query
  • Cost per API call adds up at scale
  • Web contains misinformation — requires validation
  • Rate limits cap throughput
⚠️
The practical rule: Static RAG is right for documents that do not change — company policies, technical manuals, legal contracts, historical records. Web-connected RAG is right for queries that require knowledge from today, this week, or this month. Most production systems need both, routed by query type.

The Web Search API Landscape
(What Actually Powers Live Retrieval)

The simplest way to connect RAG to the internet is through a web search API — a service that accepts a natural language query and returns structured, LLM-ready content. Not raw HTML. Not a list of links. Pre-extracted, clean text formatted specifically for injection into an LLM context window.

The landscape in 2026 has consolidated around a few major players. The right choice depends on whether you need speed, semantic understanding, full-page extraction, or cost optimization.

Latency matters for interactive agents. An independent 2026 benchmark showed Tavily averaging ~998ms while Perplexity's Sonar layer averaged 11+ seconds per query. For any user-facing application, 11 seconds per web search call is not viable. For batch research pipelines, it might be acceptable. Always measure under your actual query distribution, not vendor-reported best-case numbers.

The simplest possible internet-connected RAG

Before we get to the sophisticated CRAG architecture, here is the minimal integration — a basic web search RAG pipeline with Tavily and LangChain:

from tavily import TavilyClient
from anthropic import Anthropic
import os

tavily = TavilyClient(api_key=os.environ["TAVILY_API_KEY"])
claude = Anthropic()

def web_rag(query: str) -> str:
    # Step 1: Search the web with the user's query
    results = tavily.search(
        query=query,
        search_depth="advanced",    # "basic" for speed, "advanced" for depth
        max_results=5,
        include_raw_content=False   # True for full page text, False for summaries
    )

    # Step 2: Format results into context — Tavily returns structured JSON
    context = "\n\n".join([
        f"Source: {r['url']}\nContent: {r['content']}"
        for r in results['results']
    ])

    # Step 3: Generate answer grounded in live web context
    response = claude.messages.create(
        model="claude-sonnet-4-6", max_tokens=1024,
        system="""Answer using ONLY the provided web search results.
Cite sources with [Source: URL] after each factual claim.
If results don't contain the answer, say so explicitly.""",
        messages=[{
            "role": "user",
            "content": f"Web search results:\n{context}\n\nQuestion: {query}"
        }]
    )
    return response.content[0].text

# Example:
# query = "What is the current price of NVIDIA stock?"
# → Tavily searches live web → Claude answers from retrieved content
# → No stale knowledge, sources cited

CRAG — Corrective RAG with Web Fallback

Naive web RAG hits the internet on every query. That is expensive, slow, and unnecessary for queries your internal knowledge base handles perfectly well. Corrective RAG (CRAG) is the smarter architecture: retrieve from internal sources first, evaluate the quality of what you got, and only fall back to web search when internal retrieval is insufficient.

The CRAG loop has four steps: retrieve, grade, decide, generate. If the grader decides retrieved chunks are relevant and sufficient, answer from internal context. If not — if context is missing, outdated, or low quality — the system rewrites the query, searches the web, and augments or replaces the internal context.

CRAG architecture — corrective retrieval with web fallback
from langchain_community.retrievers import TavilySearchAPIRetriever
from langchain_anthropic import ChatAnthropic
from langchain.schema import Document
import json

llm = ChatAnthropic(model="claude-sonnet-4-6")
web_retriever = TavilySearchAPIRetriever(k=5)

def grade_relevance(query: str, docs: list[Document]) -> float:
    """LLM-as-judge: scores how relevant retrieved docs are (0-1)."""
    context = "\n---\n".join(d.page_content for d in docs[:3])
    response = llm.invoke(f"""Rate how well these documents answer this question.
Return JSON only: {{"score": 0.0-1.0, "reason": "brief explanation"}}

Question: {query}

Documents:
{context}""")
    try:
        result = json.loads(response.content)
        return result["score"]
    except:
        return 0.5   # fallback: treat as uncertain

def crag_retrieve(query: str, internal_vectorstore) -> tuple[list, str]:
    """
    CRAG pipeline: internal first, web fallback if quality is low.
    Returns (docs, source) where source is "internal" or "web"
    """
    # Step 1: Internal retrieval
    internal_docs = internal_vectorstore.similarity_search(query, k=5)

    # Step 2: Grade relevance
    score = grade_relevance(query, internal_docs)
    print(f"Internal retrieval score: {score:.2f}")

    if score >= 0.7:
        # Internal context is sufficient — no web call needed
        return internal_docs, "internal"

    # Step 3: Score too low — rewrite and search web
    rewrite_prompt = f"Rewrite for web search: '{query}'. Return only the improved query."
    web_query = llm.invoke(rewrite_prompt).content.strip()
    web_docs = web_retriever.invoke(web_query)

    # Optional: augment internal with web, or replace entirely
    if score >= 0.3:  # partial match: augment
        return internal_docs[:2] + web_docs[:3], "augmented"
    else:              # very low: replace with web
        return web_docs, "web"
💡
Why CRAG beats always-web: A well-tuned internal knowledge base answers 70–80% of queries without a web call. CRAG reduces web API costs by that proportion while still providing live data for the queries that genuinely need it. IBM's production CRAG implementation with Tavily showed that source verification — checking whether web results were actually relevant to the internal domain — eliminated a significant category of off-topic web context from entering the LLM.

Adaptive RAG — Route by Query Type

CRAG reacts to retrieval quality after the fact. Adaptive RAG is proactive: it classifies the query before retrieval and routes to the right source from the start. This eliminates unnecessary retrieval attempts on queries that clearly need live data, and unnecessary web API calls on queries that clearly do not.

from anthropic import Anthropic
import json, re
from datetime import datetime

client = Anthropic()

FRESHNESS_KEYWORDS = [
    "current", "latest", "today", "now", "recent", "2026",
    "this week", "yesterday", "just announced", "breaking",
    "price", "stock", "news", "update", "new release"
]

def classify_query(query: str) -> str:
    """
    Classify query into routing category:
    - "static":     internal KB sufficient (policies, manuals, historical facts)
    - "live":       requires current web data (prices, news, recent events)
    - "uncertain":  unclear — use CRAG (internal first, web fallback)
    """
    query_lower = query.lower()

    # Fast rule-based pre-check (no LLM cost)
    if any(kw in query_lower for kw in FRESHNESS_KEYWORDS):
        return "live"

    # LLM classifier for ambiguous cases
    response = client.messages.create(
        model="claude-sonnet-4-6", max_tokens=100,
        messages=[{
            "role": "user",
            "content": f"""Classify this RAG query. Return JSON only:
{{"route": "static|live|uncertain", "reason": "one sentence"}}

"static" = can be answered from internal documents (policies, facts, docs)
"live" = requires current web data (prices, news, events after {datetime.now().year - 1})
"uncertain" = unclear, try internal first

Query: {query}"""
        }]
    )

    try:
        result = json.loads(response.content[0].text)
        return result["route"]
    except:
        return "uncertain"

def adaptive_retrieve(query: str, vectorstore, web_retriever) -> list:
    route = classify_query(query)
    print(f"Route decision: {route}")

    if route == "static":
        return vectorstore.similarity_search(query, k=5)

    elif route == "live":
        return web_retriever.invoke(query)

    else:  # uncertain → CRAG
        return crag_retrieve(query, vectorstore)[0]

# Examples:
# "What is our refund policy?" → static (internal KB)
# "What is GPT-5's context window?" → live (web search)
# "How does our pricing compare to competitors?" → uncertain (CRAG)

Tiered Freshness Strategy — Managing Cost and Latency

Hitting a web search API on every live query is expensive at scale. The solution is a tiered caching strategy based on how quickly information in each category becomes stale.

Content Type Staleness Rate Cache Duration Retrieval Strategy Example Queries
Real-time Changes hourly No cache Always live web Stock prices, live sports scores, breaking news
Daily-fresh Changes daily Cache 4–24h Web + TTL cache Today's news, regulatory announcements
Semi-stable Changes weekly Cache 24–72h Cached web results Product specs, API docs, company profiles
Stable Changes monthly+ Internal KB Static RAG only Legal contracts, manuals, historical facts 
import redis
import hashlib
from tavily import TavilyClient

r = redis.Redis(host="localhost", port=6379)
tavily = TavilyClient(api_key="...")

def cached_web_search(query: str, ttl_hours: int = 24) -> list:
    """Web search with Redis caching. TTL varies by content type."""
    cache_key = f"rag:web:{hashlib.md5(query.encode()).hexdigest()}"

    # Check cache first
    cached = r.get(cache_key)
    if cached:
        print("Cache hit — no web API call")
        import json
        return json.loads(cached)

    # Cache miss — hit live web
    results = tavily.search(query=query, max_results=5)["results"]

    # Cache with appropriate TTL
    r.setex(cache_key, ttl_hours * 3600, json.dumps(results))
    print(f"Web search complete, cached for {ttl_hours}h")
    return results

# Usage:
# Real-time: cached_web_search(query, ttl_hours=0)  # no cache
# Daily news: cached_web_search(query, ttl_hours=4)
# Semi-stable: cached_web_search(query, ttl_hours=48)
# This cuts web API calls by 60-70% for typical enterprise workloads

The most common mistake I see teams make when adding internet connectivity to RAG is treating it as an all-or-nothing decision. They either use static RAG for everything (and wonder why it fails on time-sensitive queries) or route every query through a web search API (and then complain about latency and cost). The correct architecture is surgical: classify your queries by freshness requirement, use internal retrieval for the 70–80% that do not need live data, and reserve web search for the minority that genuinely do. CRAG is the safety net for the uncertain middle. That tiered approach is what separates a production-grade internet-aware RAG from an expensive demo.

Three Things to Take Away

First: static RAG has a knowledge decay problem that matters at different rates for different query types. Stock prices decay in minutes. News decays in hours. Product specs decay in days. Company policies decay in months. Your routing architecture should reflect this — do not treat all queries the same.

Second: CRAG is the architecture for most production systems. It provides the cost and speed advantages of internal retrieval for the majority of queries, while automatically falling back to live web data when internal retrieval quality is insufficient. The LLM-as-judge grader is the key component — it makes the fallback decision data-driven rather than rule-based.

Third: semantic caching is the cost control lever. Caching web search results with appropriate TTLs by content type can cut web API costs by 60–70% for typical enterprise workloads. Real-time data skips the cache. Semi-stable product information caches for 48 hours. Stable reference documents never hit the web at all. The tiered strategy is what makes live web retrieval economically viable at scale.

The combination of adaptive query routing, CRAG fallback logic, and tiered caching is what "connecting RAG to the internet" actually means in a production system. Each piece is individually simple — together they form a retrieval architecture that is both accurate and economically sustainable.

Your next step
Add a Tavily or Serper web search retriever to your existing RAG system this week. Do not replace your internal retrieval — add it as a CRAG fallback triggered when your relevance grader scores below 0.7. Measure the proportion of queries that trigger the web fallback. If it is above 30%, your internal knowledge base needs more comprehensive coverage. If it is below 5%, your queries are mostly stable content and you may not need live web retrieval at all.
